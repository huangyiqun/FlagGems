# DeepSeek-V4 Attention Verification

This directory contains local verification scripts for
`runtime/backend/_nvidia/fused/dsv4_attention_triton.py`.

The implementation must use only FlagGems code. vLLM is used only to generate
offline oracle tensors, never as an implementation dependency.

## Environment Model

Use two conda environments:

- `vllm0.20.2`: generates oracle `.pt` files from the current vLLM DeepSeek-V4
  operators.
- `vllm0.17`: runs the FlagGems implementation and compares it against the
  saved oracle data.

Every script prints the active conda env, Python executable, torch version,
CUDA version, and CUDA availability. Treat that output as part of the test:
it catches accidentally running a script in the wrong environment.

## Generated Oracle Files

`dsv4_attention_oracle.pt`

- Generated by `verify_dsv4_oracle_gen.py` in `vllm0.20.2`.
- Contains vLLM oracle inputs and outputs for RMSNorm, RoPE plus FP8 KV cache
  insert, KV cache gather/dequantization, sparse prefill, sparse decode with
  extra cache, and top-k/SWA index utilities.

`dsv4_fp8_einsum_oracle.pt`

- Generated by `verify_dsv4_fp8_einsum_oracle_gen.py` in `vllm0.20.2`.
- Contains vLLM oracle inputs and outputs for
  `torch.ops.vllm.deepseek_v4_fp8_einsum`.
- Covers contiguous scale layout, vLLM-strided scale layout, a larger
  DeepSeek-like layout, and the layout produced by vLLM inverse-RoPE FP8
  quantization.

## Scripts

`verify_dsv4_oracle_gen.py`

- Purpose: generate `dsv4_attention_oracle.pt` with vLLM 0.20.2.
- Uses vLLM custom operators as the source of truth.
- Expected result: the script prints environment information and ends with:

```text
wrote .../dsv4_attention_oracle.pt
```

`verify_dsv4_subops.py`

- Purpose: verify individual FlagGems DeepSeek-V4 sub-operators against the
  vLLM oracle.
- Covers:
  - `dsv4_fused_q_kv_rmsnorm`
  - `dsv4_qnorm_rope_kv_rope_quant_insert`
  - `dsv4_dequantize_and_gather_k_cache`
  - `dsv4_compute_global_topk_indices_and_lens`
  - `dsv4_combine_topk_swa_indices`
- Expected result:

```text
PASS rmsnorm_vs_vllm
PASS qnorm_cache_gather_vs_vllm
PASS indices_vs_vllm
```

`verify_dsv4_prefill.py`

- Purpose: verify sparse MLA prefill behavior.
- Compares both the public FlagGems sparse MLA wrapper and the local
  `dsv4_flash_mla_sparse_prefill` wrapper against the vLLM oracle.
- Covers:
  - `d_qk=512`, 64 heads
  - `d_qk=576`, 128 heads, no optional `attn_sink` or `topk_length`
  - `d_qk=576` with `topk > skv`, including invalid indices
- Expected result:

```text
PASS raw_flaggems_prefill_512_h64_vs_vllm
PASS raw_flaggems_prefill_576_h128_no_optional_vs_vllm
PASS prefill_512_vs_vllm
PASS prefill_576_h128_no_optional_vs_vllm
PASS prefill_576_topk_gt_skv_vs_vllm
```

`verify_dsv4_decode.py`

- Purpose: verify the local sparse decode path against vLLM
  `flash_mla_with_kvcache` oracle output.
- Covers:
  - FP8 DeepSeek-V4 MLA KV cache dequantization
  - primary sparse cache indices
  - extra compressed cache indices
  - `topk_length`, `extra_topk_length`
  - invalid indices
  - `attn_sink`
- Expected result:

```text
PASS decode_sparse_with_extra_cache_vs_vllm
```

`verify_dsv4_e2e.py`

- Purpose: verify the public composition entry point
  `dsv4_attention_triton`.
- Covers:
  - prefill wrapper dispatch and in-place output write
  - decode wrapper dispatch and in-place output write
- Expected result:

```text
PASS e2e_prefill_wrapper_vs_vllm
PASS e2e_decode_wrapper_vs_vllm
```

`verify_dsv4_fp8_einsum_oracle_gen.py`

- Purpose: generate `dsv4_fp8_einsum_oracle.pt` with vLLM 0.20.2.
- Uses `torch.ops.vllm.deepseek_v4_fp8_einsum` as the oracle.
- Expected result: the script prints environment information and ends with:

```text
wrote .../dsv4_fp8_einsum_oracle.pt
```

`verify_dsv4_fp8_einsum.py`

- Purpose: check whether existing FlagGems FP8 block matmul kernels can
  reproduce the vLLM DeepSeek-V4 FP8 einsum contract
  `bhr,hdr->bhd` with recipe `[1, 128, 128]`.
- Compares:
  - public `flag_gems.w8a8_block_fp8_matmul`
  - Hopper backend `w8a8_block_fp8_matmul`
  - vLLM oracle output
  - an FP32 diagnostic reference
- Expected result: each case prints `PASS`, including:

```text
PASS synthetic_contiguous_scales_b7_g2_r512_d384
PASS synthetic_vllm_stride_scales_b9_g3_r1024_d512
PASS synthetic_deepseek_like_vllm_layout_b5_g2_r8192_d1536
PASS inv_rope_quant_layout_b11_g2_r1024_d384
```

## Typical Run Sequence

From `/workspace/FlagGems-dev`:

```bash
source /root/miniconda3/etc/profile.d/conda.sh

conda activate vllm0.20.2
python src/flag_gems/runtime/backend/_nvidia/fused/verify/verify_dsv4_oracle_gen.py
python src/flag_gems/runtime/backend/_nvidia/fused/verify/verify_dsv4_fp8_einsum_oracle_gen.py

conda activate vllm0.17
python -m py_compile \
  src/flag_gems/runtime/backend/_nvidia/fused/dsv4_attention_triton.py \
  src/flag_gems/runtime/backend/_nvidia/fused/verify/*.py
python src/flag_gems/runtime/backend/_nvidia/fused/verify/verify_dsv4_subops.py
python src/flag_gems/runtime/backend/_nvidia/fused/verify/verify_dsv4_prefill.py
python src/flag_gems/runtime/backend/_nvidia/fused/verify/verify_dsv4_decode.py
python src/flag_gems/runtime/backend/_nvidia/fused/verify/verify_dsv4_e2e.py
python src/flag_gems/runtime/backend/_nvidia/fused/verify/verify_dsv4_fp8_einsum.py
```

## Expected Precision

Exact checks are used where vLLM and FlagGems should be bit-identical:

- RMSNorm output in the saved test case
- FP8 KV cache bytes
- dequantized gather output
- top-k/global index tensors and lengths

Approximate checks are used for sparse MLA and FP8 matmul outputs:

- prefill sparse MLA: `atol=3e-2`, `rtol=3e-2`
- decode sparse MLA and e2e wrapper output: `atol=4e-2`, `rtol=4e-2`
- FP8 einsum checks: `atol=4e-2`, `rtol=4e-2`

These tolerances are intended to catch behavioral mismatches while allowing
normal BF16/FP8 accumulation differences between kernels.

## Notes

- The oracle generation scripts intentionally import vLLM and call vLLM custom
  operators.
- The verification scripts should import FlagGems implementation code for the
  tested path. They should not call vLLM operators during validation.
- `verify_dsv4_fp8_einsum.py` uses an FP32 `torch.einsum` only as a diagnostic
  reference for reporting error magnitude. It is not part of
  `dsv4_attention_triton.py`.
